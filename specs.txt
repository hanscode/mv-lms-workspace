Project Overview
The purpose of this project is to rebuild the McKinney-Vento training site. The training site will be custom-built, removing the restrictions and structural necessities that WordPress implements, and will be focused on the following goals: 

To make onboarding and access to courses as easy as possible for all users, whether it’s their first time or they’re a returning user. 
To remove all “stop points” from the user experience, so that there is never a moment the user isn’t sure what to do next. 
Improve reporting for both MV Administrators and Liaisons. 
Reduce administrative complexity. 

The language framework will be determined once the specification has been approved.
Leadership Notes
We have an interesting challenge with the learning platform based on what we’ve learned over the course of the last few years. We want to build a platform that will allow for easy registration from essential staff, with the least amount of steps possible, which doesn’t require email validation/verification, domain verification, or a complex registration, and make sure the solution doesn’t add-in additional complexity or time during the launch process, or require intervention from the support team. Finally, we want to do our best to prevent unauthorized users or non-paying users from accessing the system.

The solution I’m proposing, detailed below, while it initially may sound complex, is aimed at resolving all of these issues and making the registration process as streamlined as possible. 

Major points: 

Will not require email verification for registration. 
Passwordless system that will use magic link email to login.
No email domain verification during registration, users can use any email address they have access to.
Each school district, region and state will have a unique registration URL, with a path to easy registration for the user, regardless of role. 
No longer need separate registration forms for free liaisons or substitute teachers. 


Training Portal
Compatibility & Accessibility
The training portal will be designed and developed with both accessibility and compatibility in mind. The platform will be kept in compliance with the ADA requirements for websites and platforms.

The portal will also be available on both desktop and mobile, ensuring that staff without access to a work or personal computer will be able to complete the training if they have a mobile device. 
Registration Pages
The training portal will not have a “register page” or “register link” available from the main page. On creating a state, region or school district within the administrative back-end, a unique URL will be generated, for example: 

https://training.mckinney-venro.org/jfo12

This link is provided to the client who is purchasing the training during the launch process who can then share this link with their liaisons and/or essential staff, and the interface will be dependent on the location tier that the link is generated from. 

The design of these pages will be focused on making every choice and action deliberate and obvious, with guided indicators letting users know exactly what they need to do through each step of the process, and what to do if they’re unable to take an action.

Registration Page Notes
The registration page links are downward compatible. Meaning if our client is at the state, they can send their regional coordinates the state link, and regional coordinators can get their region registration by selecting their region and sending that link out to liaisons. Liaisons can follow the same process, by selecting their school district from the region page, and copying that link to all of their staff.

The system will also allow quick export to CSV of all registration links within a state/region that can be sent to state or regional coordinators to share with their liaisons. 
User Login
User login will be done via a magic link, and will not require a password. When returning to the portal platform, they will be able to enter their email address from the main page. 

They will receive the following message:

“We’ve just sent you an email with a link to login. Click the login button in this email to access the training portal.”

After clicking on the link in the email the user will be redirected to their learning dashboard.

The token in the email will also not expire for 30 days, and will not expire after use, meaning even if their security software scans the link it will still be usable when the user clicks it. 
Learning Dashboard
The learning dashboard will be separated into two sections. The first section will include courses specific to the user’s role, and are courses required for the user to complete their required training and stay in compliance with the McKinney-Vento act. This will be clearly marked and described in the course section.

Below this, will be a list of optional courses. These are not courses for other roles, but additional courses MV provides and may provide in future to add additional value to our training for respective roles. If there are no additional courses available, this section will be hidden.

The display of courses will be determined during the design phase but will likely mirror the updated design of our learn page on the current WordPress site and will additionally display the time expected to complete the course. 

A completed course will show a completion ribbon on the course as well as a link to download the completion certificate. 

Displayed Courses

The courses available to the user depend on the settings of the client and any other clients above it in the Option hierarchy. 

If client specific courses is not toggled for the client, or any clients above it in the hierarchy, the user will see any mandatory and optional courses including courses that are marked client specific. 

If client specific courses is toggled for the client, or any clients above it in the hierarchy, the user will only see the selected client specific courses.

If the client and the client above it in the Option hierarchy have a different selection of client specific courses, the user will see client specific courses from both clients. 

If there is a mix, i.e. the client has client specific courses but a parent client (above it in the Option hierarchy) does not have client specific courses toggled, the user will see all non-client specific optional and mandatory courses and the client specific courses selected for the client.   
Taking a Course
By clicking on a course the user will be brought through the course content. Courses will be broken down into lessons and each lesson will be made up of sections (section types described in administration section), and the title of the lesson will be displayed along with the content. At the top of the course interface will be a navigation guide, showing the user the amount of lessons remaining to complete the course. 

At the end of the course, the user will be automatically emailed their completion certificate as well as offered a download button to download their completion certificate. They can then return to the learning dashboard. 

Courses will be role-limited, so only courses relevant to the role of the user, as well as optional courses based on client access will be displayed. 

Course Timer

The LMS should track the amount of time a user spends taking a course. This will be a timer that runs in the background when a user is taking a course. The total time it takes a user to complete a course should be displayed on the certificate, and the total time should show up on the reports once the course is completed. 

This is for school districts that award staff Personal Development credits, some teachers unions have negotiated anything taking more than 3 hours they should receive credit for, so this would apply to our liaison courses (and future staff courses).

Program Creation
The program management interface will allow a member of the support team to create a program for a state, region, district or school. A program can include users/trainees from one or more districts, cross districts or schools, depending on the user list provided during program creation. 

A program will be created in 4 steps. After each step, the program creation process will be saved on the backend so the user can pause and resume program creation. This is to allow users flexibility in program creation, lessen front end resources, and allow for recovery in times of technical failure. 

The 4 steps to create a program is as follows:   

Step1: Client Selection & Program Name

From a dropdown list of existing clients in the system, the user will select the client that the program should be associated with. Only active clients will be able to be selected from this dropdown. 

They will then be able to name the program. This is important as some clients may wish to run multiple programs (for each district) to keep information and reporting separate. 

Step 2: User Import

Once a client is selected, the upload interface will be displayed for the user to upload a list of users who will be included in the program, this can include both existing users and users that require registration. 

The CSV template to be used for user import can be downloaded from within this interface and will provide columns for the following fields: 

First Name
Last Name
Email Address
MV Option ID
Program Leader (Yes/No)

When the import file is uploaded, the system will perform the following checks: 

Determine which users, by email address, are already users in the system. For existing users, it will confirm whether the user is already a user within the option ID referenced. If not, it will move the user to that option ID. 
Determine which users, by email address, need to register on the training site. The system will create a user account and place them in the appropriate option. 

Note: Users designated as a program leader must be in a higher tier option than all users that are included in the program, this will generally mean the user must be a regional coordinator or district liaison.
